# 高级用法：自定义字体 / 字号 / 修改译文

面向想自行调整汉化包的高级用户。所有工具在 `tools\`，需要 Python 3 + `pip install pillow numpy`，命令建议用 `py -X utf8` 运行避免控制台编码报错。

## 1. 更换字体

游戏全部界面/字幕只用一个字体包 `GameData\Textures\orbitfonts.utx`，用 `font_gen.py` 可以从任意 TTF/OTF 重新生成：

```
py -X utf8 tools/make_charset.py charset.txt
py -X utf8 tools/font_gen.py --font 你的字体.ttf --charset charset.txt --out orbitfonts.utx
```

生成后覆盖 `GameData\Textures\orbitfonts.utx` 即可，无需其他改动。

对字体的要求：
- **必须覆盖你所用文本的全部汉字**（推荐至少 GB2312 范围）。缺字会显示为空白（回落到空白字形）
- 授权允许再分发才可以打包发布（思源黑体/更纱黑体等 OFL 字体可以；**微软雅黑等 Windows 自带字体不可以**）
- 粗体字形在小字号（13px 行高）下更清晰，原版风格也是 Bold

### 1.1 混合英文原版 Latin（可选）

默认生成器会用 `--font` 指定的字体绘制全部字符。若想让英文 ASCII 保持原版游戏风格，可以额外传入英文版 `orbitfonts.utx`：

```
py -X utf8 tools/font_gen.py --font 字体.ttf --charset charset.txt --latin-source "..\.originalbackup\GameData\Textures\orbitfonts.utx" --out orbitfonts.utx
```

该模式只复用英文版 `0x20-0x7E` 可打印 ASCII 字形；中文、全角标点和其他 GBK 字符仍由 `--font` 渲染。未传 `--latin-source` 时仍使用原先的纯 TTF/OTF 生成管线。

## 2. 调整字号

游戏按用途固定使用 5 个字号对象（OrbitBold8/12/15/18/24），字号↔字形行高（像素）的默认映射取自官方日文版：

| 字号对象 | 默认行高 | 主要用途 |
|---|---|---|
| OrbitBold8 | 13 | 菜单按钮、小字 UI |
| OrbitBold12 | 16 | 一般 UI |
| OrbitBold15 | 20 | 字幕 |
| OrbitBold18 | 24 | 标题类 |
| OrbitBold24 | 29 | 大标题 |

想整体放大缩小某档，用 `--cells` 覆盖（只写想改的档位）：

```
py -X utf8 tools/font_gen.py --font 字体.ttf --charset charset.txt --cells "15=22,8=14" --out orbitfonts.utx
```

注意：UI 布局按行高排版，行高改动过大会导致文字挤压或溢出边框；建议每档 ±2px 内微调。

## 3. 修改译文

文本分两类，都必须以 **GBK 编码、无 BOM** 保存（用记事本"ANSI"或编辑器选 GBK；**不要存成 UTF-8**，会显示乱码）：

### 3.1 字幕 / 提示 / 关卡文本（`GameData\System\*.int`）

普通 ini 风格键值，直接编辑等号右边的值即可，例如 `subtitles_geo_01.int`：

```ini
[SubtitleEvents]
GEO01_Boss_066=博斯：中文字幕内容
```

注意保留 `%k %o %s` 之类的占位符（运行时替换为按键名等）。

### 3.2 菜单文字（优先改 `GameData\System\XinterfaceCtmenus.int`）

菜单文字可以通过 `XinterfaceCtmenus.int` 覆盖，格式仍是 ini 风格。官方日文包已使用此文件，Steam 英文版 + Fix 环境下也已实测可生效：

```ini
[CTAcceptSettingsTimedMenu]
Label.Text="使用这些设置？"
ALabel.Text="是："

[CTGraphicsOptionsPCMenu]
Options[0].Items=("最低","低","中","高","最高","自定义")
```

这一路径比 `PropertyOverrides` 更适合发布包，因为不需要复制完整 UE 结构体，也不容易丢坐标、字体、样式等字段。

`PropertyOverrides` 仍可作为高级 fallback：文件名必须是 `包名.类名.txt`（如 `XInterfaceCTMenus.CTMenuMain.txt`），内容是 defaultproperties 风格的整行覆盖：

```uc
MenuOptions(0)=(Blurred=(Text="新的战役",PosX=0.68375,PosY=0.48),BackgroundBlurred=(PosX=0.68375,PosY=0.48,ScaleX=0.38,ScaleY=0.04333),OnSelect="NewGameSelected",Style="ButtonTextStyle1")
```

使用 `PropertyOverrides` 时必须整个结构体一起写（从 `reference\export\` 里对应类的 .uc 源码复制原值，只改 `Text=`/`HelpText=` 字段），只写 Text 会丢掉坐标等其他字段。加载失败会静默跳过——改完进游戏看不到效果，先检查文件名与结构体拼写。

### 3.3 用了新汉字之后

译文里出现字体包没有的字会显示空白。改完译文建议重跑字符集扫描+字体生成（发布版字符集就是从全部译文扫描出来的）；开发期偷懒可以直接用 `make_charset.py` 的 GB2312 全集，覆盖绝大多数简体用字。

### 3.4 重新生成翻译 JSON / catalog

正式翻译表由 `tools\make_translation_json.py` 维护。它用日文包的 `.int` 作为“哪些文本需要覆盖”的索引，再从 Steam 英文版 + Fix 的 `GameData` 里追溯英文 source，生成：

- `translation.json`：给译者编辑的 UTF-8 JSON，只含 `note`、`en`、`jp`、`de`、`zh_CN`
- `reference\export\localization_catalog.json`：机器使用的 catalog，记录输出文件、section/key、value 类型、模板、控制行
- `reference\export\localization_audit.json`：生成审计和统计
- `reference\export\localization_jp_only_skipped.json`：暂未进入 catalog 的 JP-only/skipped 明细

译者须知：`.int` 成品文本必须用 GBK，但 `translation.json` 必须保存为 UTF-8。不要把两者混用：如果 `translation.json` 被保存成 GBK/ANSI，或 JSON 语法损坏、schema 不匹配，生成器会硬失败并拒绝覆盖；每次成功生成前会自动把旧 `translation.json` 备份为 `translation.json.bak`，用于保护尚未提交的译文。

常用命令：

```powershell
py -X utf8 tools/make_translation_json.py
py -X utf8 tools/make_translation_json.py --allow-skipped
py -X utf8 tools/make_translation_json.py --reset-translations --allow-skipped
```

参数语义：
- 默认会读取现有 `translation.json`，按 `(group, note, en)` 匹配旧条目并保留已有 `zh_CN`；source 修正后可以安全重跑
- `--reset-translations` 会显式清空已存在的 `zh_CN`，只用于从零重建翻译表
- 默认 skipped 严格失败，防止漏项静默进入正式翻译；审查期需要生成当前过渡版 JSON 时才加 `--allow-skipped`
- 默认会临时调用 `build_langpack.py` 做构建校验；验证输出写入临时目录，不会清理仓库 `build\`；只想检查 JSON 生成流程时可加 `--skip-build-validation`

source 追溯范围包括 `GameData\System\*.int`、`GameData\System\*.u`、`GameData\Properties\*.u`、`GameData\Maps\*.ctm`，以及可选的 `TranslationPackFormOnline\German Files\*.det`（德语参照）。其中 `voicepacks.int` 来自 `GameData\Properties\voicepacks.u` 的 class default，但仍输出为 `GameData\System\voicepacks.int`。通用 `.int` 生成会为 EN-only 键补 passthrough literal，确保生成文件覆盖被替换英文文件的所有运行时键；passthrough 会做与英文 fallback 相同的最小 GBK 规范化。工具运行时会导出必要 `.uc` 到 `reference\export\包名\` 供追溯；除已保留的菜单参照源外，后续批量导出的 `.uc` 是中间噪音，默认不需要提交。

英文、日文、德语均相同的条目会合并到 `shared.enjp`，同一 `(en, jp, de)` 组合只能填写一份 `zh_CN`，适合全局复用；如果后续 source 变化导致条目离开 shared，生成器会按 `(en, jp, de)` 回填旧 shared 译文。

### 3.5 从翻译 JSON 生成文本包

`tools\build_langpack.py` 使用“机器 catalog + 用户翻译 JSON”生成文本包。用户 JSON 只保留分组、id、note、en、jp、de、zh_CN；路径、键名、控制行、菜单数组等构建细节由 catalog 管理。

常用命令：

```powershell
py -X utf8 tools/build_langpack.py
py -X utf8 tools/build_langpack.py --json translation.json --catalog reference/export/localization_catalog.json
py -X utf8 tools/build_langpack.py --out D:\tmp\swrc-build
```

默认要求所有 `zh_CN` 非空。开发期只想抽样构建时可以显式允许空译文回退英文：

```powershell
py -X utf8 tools/build_langpack.py --allow-untranslated
```

这只是开发用逃生口，发布前必须不用此参数完整构建。工具默认只写仓库 `build\`，也可用 `--out` 指向临时目录；不会直接修改游戏目录，发布或本地覆盖时再复制输出目录下的 `GameData\`。

构建器会做这些安全检查：
- `translation.json` 与 catalog schema 必须匹配，JSON 内不能有重复 key，且不能有 catalog 未引用的多余翻译条目
- `zh_CN` 的 printf 占位符类型/顺序必须和英文 source 一致
- 正式译文必须 GBK 可编码；`--allow-untranslated` 的英文 fallback 会做最小兼容替换（例如 `©` → `(C)`），但不会放宽正式中文译文
- 输出路径限制在输出目录的 `GameData\System\*.int`，并通过该目录的 `manifest.json` 管理清理范围，避免误删非本工具产物

## 4. 查看 / 反解字体包（调试用）

```
py -X utf8 tools/swrc_package.py orbitfonts.utx                     # 导出表概览
py -X utf8 tools/swrc_package.py orbitfonts.utx font OrbitBold15   # 字形数/页数/键分布
py -X utf8 tools/swrc_package.py orbitfonts.utx tex OrbitBold15_Page00          # 贴图页信息
py -X utf8 tools/swrc_package.py orbitfonts.utx tex OrbitBold15_Page00 页.png   # 贴图页转 PNG（输出文件可选）
```

## 5. 依赖关系提醒

- CJK 能显示的前提是 **Fix（≥ 含 CJK 渲染支持的版本）的 Mod.dll**——字体和文本只是数据，双字节配对逻辑在 Mod.dll 里
- Steam"验证游戏完整性"会还原被覆盖的原版文件，验证后需重装汉化包
